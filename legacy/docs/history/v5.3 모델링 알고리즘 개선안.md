
## 1. 검토 배경

AI 에너지 사용 예측 페이지에 적용된 v5.3 모델은 기상변수, 생산실적, 재공품 실적 등을 기반으로 일별 전력, 연료, 용수 사용량의 예측값과 정상범위 밴드를 산출한다.

운영 목적은 단순 예측 정확도 향상에 그치지 않고, 정상 가동 조건에서 기대되는 적정 사용량 범위를 제시한 뒤 과사용 의심 일자를 식별하고, 추후 관리자에게 이상원인을 보고받기 위한 것이다.

이번 검토의 핵심 질문은 다음과 같다.

- 왜 quantile regression의 범위가 P05~P95인지
- P95 값이 어떤 원리로 산출되는지
- 정상 가동 상태인데도 P95를 벗어나는 경우가 많은 이유
- 현재 알고리즘이 과사용 탐지 목적에 적합한지
- 논리적 결함 또는 개선점이 있는지

## 2. v5.3 모델 구조 요약

v5.3은 분위수 회귀 기반 예측구간 모델이다. 각 공장과 대상 항목별로 P05, P50, P95를 각각 예측한다.

대상 항목은 다음 3개다.

- 전력: 전력량[kWh]
- 연료: 연료량[Nm3]
- 용수: 용수량[ton]

모델은 LightGBM, XGBoost, CatBoost를 사용하며, 각 알고리즘을 분위수별로 따로 학습한다. v5.3 기준으로는 M1, M3, M4의 3개 피처 관점이 사용되고, 각 분위수당 9개 모델이 앙상블된다.

즉 한 공장-타깃 조합마다 다음 구조가 된다.

- P05 모델 9개
- P50 모델 9개
- P95 모델 9개
- 총 27개 모델

학습 타깃은 원 사용량이 아니라 `log1p(사용량)` 공간에서 학습되고, 예측 후 `expm1`로 원래 단위로 되돌린다.

## 3. 왜 P05~P95 범위인가

코드상 분위수는 `0.05, 0.50, 0.95`로 고정되어 있다.

이는 중앙 90% 예측구간을 의미한다. 즉 모델이 같은 조건이라고 판단하는 날짜들 중 약 90%의 사용량이 들어올 것으로 기대하는 범위다.

중요한 점은 P95가 정상과 비정상을 나누는 절대 경계가 아니라는 것이다.

이론적으로 밴드가 잘 보정되어 있어도 정상 데이터의 약 5%는 P95보다 클 수 있다. P05~P95 구간은 정상 운전의 모든 경우를 포함하는 관리 기준선이 아니라, 통계적으로 자주 관찰되는 중심 범위에 가깝다.

또한 시스템은 하루에 5개 공장 x 3개 대상, 총 15개 항목을 동시에 본다. 각 항목이 독립적으로 P95 초과 확률 5%를 가진다고 단순 가정하면, 하루에 하나 이상 P95를 초과할 확률은 다음과 같다.

```text
1 - 0.95^15 = 약 54%
```

따라서 모든 항목이 정상 가동 중이어도, 매일 여러 항목을 동시에 감시하면 P95 초과가 자주 보이는 것은 자연스러운 현상이다.

## 4. P95 값 산출 원리

P95는 P50에 일정 오차율을 더한 값이 아니다. 별도로 학습된 조건부 95분위수 모델의 예측값이다.

산출 흐름은 다음과 같다.

1. 사용량을 `log1p` 변환한다.
2. LightGBM, XGBoost, CatBoost 각각을 `alpha=0.95` 분위수 손실로 학습한다.
3. M1, M3, M4 피처 관점별 모델을 합쳐 P95 후보 모델 9개를 만든다.
4. 검증셋에서 pinball loss가 작아지도록 9개 모델의 앙상블 가중치를 최적화한다.
5. 예측 시 9개 모델의 P95 예측값을 가중 평균한다.
6. P05, P50, P95가 뒤집히지 않도록 row-wise 정렬로 단조성을 강제한다.
7. CQR 보정값 `q_hat`을 P95에 더해 최종 상한을 넓힌다.

최종 형태는 개념적으로 다음과 같다.

```text
raw_P95 = weighted_average(P95_model_1 ... P95_model_9)
final_P95 = raw_P95 + cqr_q_hat
```

P05는 반대로 `raw_P05 - cqr_q_hat`으로 낮아진다.

## 5. CQR 보정의 의미

v5.3은 Conformalized Quantile Regression(CQR)을 사용한다. CQR은 검증구간에서 실제값이 예측 밴드 밖으로 얼마나 벗어났는지를 보고, 밴드를 사후 보정하는 방식이다.

코드상 검증셋에서 다음 score를 계산한다.

```text
score = max(P05 - actual, actual - P95)
```

이 score의 90% 분위수 수준을 `q_hat`으로 삼고, 최종 밴드를 다음처럼 넓힌다.

```text
P05 = P05 - q_hat
P95 = P95 + q_hat
```

이 구조는 검증구간에서는 P05~P95 밴드가 90% coverage에 맞도록 보정하는 역할을 한다.

다만 한계도 있다.

- 검증구간이 현재 운영분포를 대표하지 못하면 보정이 틀어진다.
- 상단과 하단을 같은 폭으로 보정하므로 과사용 탐지에 특화되어 있지는 않다.
- 공장-대상별로 상단 초과가 많은 경우에도 상단만 별도 보정하지 않는다.

## 6. 실제 성능 및 운영 이력 관찰

활성 v5.3 모델의 성능 파일 기준 PICP_90 평균은 약 0.86이다. 목표는 0.90이지만 실제 테스트 구간에서는 평균적으로 86%만 밴드 안에 들어왔다.

공장-대상별로는 더 큰 편차가 있다. 일부 항목은 PICP_90이 0.67~0.77 수준까지 낮다. 이는 해당 항목의 밴드가 너무 좁거나, 최근 운영분포 변화가 충분히 반영되지 않았다는 신호다.

`prediction_log` 운영 이력 집계 결과는 다음과 같았다.

```text
전체 비교 가능 행: 1,785건
P95 초과: 108건, 6.1%
P05 미만: 120건, 6.7%
밴드 밖 전체: 228건, 12.8%
```

전체 평균만 보면 P95 초과율 6.1%는 이론적 5%보다 약간 높은 수준이다. 그러나 항목별로는 초과가 집중되는 대상이 있다.

```text
김해 연료: P95 초과 16.8%
광주 용수: P95 초과 11.8%
남양주2 용수: P95 초과 10.9%
김해 전력: P95 초과 10.1%
논산 연료: P95 초과 9.2%
```

이 항목들은 정상 가동 상태에서도 P95 초과가 자주 발생할 가능성이 높다. 따라서 우선 보정 또는 재학습 검토 대상이다.

## 7. 현재 알고리즘의 장점

현재 구조에는 장점도 분명히 있다.

- 점추정 P50뿐 아니라 정상범위 밴드를 제공한다.
- P05/P95를 별도 모델로 학습해 사용량 변동 폭을 조건별로 다르게 표현할 수 있다.
- CQR 보정으로 검증구간 coverage를 맞추려는 장치가 있다.
- 당일 타깃 실측 누수를 막기 위해 lag1, r7mean, intensity_lag1 등은 shift 처리되어 있다.
- 생산량, 기상, 휴일, 재공품, 보틀링, 활성 SKU 등 현장 요인을 반영하려는 방향은 적절하다.
- 배치 예측에서 미래 날짜의 P50을 다음날 lag로 되먹이는 recursive lag 구조가 있어 다중일 예측 시 lag 동결 문제를 완화한다.

## 8. 주요 문제점

### 8.1 P95 초과를 이상으로 직접 해석하는 문제

P95 초과는 통계적으로 상위 5% 영역이라는 뜻이지, 곧바로 비정상 과사용이라는 뜻은 아니다.

정상 가동 중에도 고생산, 특정 제품 mix, 세척, 재가동, 날씨 급변, 라인 전환 등으로 P95를 넘을 수 있다.

따라서 P95 초과는 이상 보고 대상이라기보다 1차 후보로 보는 것이 더 적절하다.

### 8.2 다중 비교 문제

하루에 15개 항목을 동시에 감시하면 정상 상태에서도 하나 이상 P95 초과가 발생할 확률이 높다.

현재 UI나 보고 로직이 이 다중 비교 효과를 충분히 반영하지 않으면, 관리자 입장에서는 정상일에도 이상 알림이 너무 자주 발생한다고 느낄 수 있다.

### 8.3 일부 항목의 under-coverage

운영 이력에서 김해 연료, 광주 용수, 남양주2 용수, 김해 전력은 P95 초과율이 10%를 넘는다.

이는 P95 상한이 해당 항목의 실제 상단 분포를 충분히 포착하지 못하고 있음을 의미한다.

### 8.4 CQR 보정의 상하 대칭성

현재 CQR은 하나의 `q_hat`으로 P05와 P95를 같은 폭만큼 넓힌다.

하지만 운영 목적은 주로 과사용 탐지이므로 상단 보정 품질이 더 중요하다. 하단과 상단의 오차 분포가 다르면 단일 대칭 보정은 비효율적이다.

### 8.5 검증구간 의존성

CQR은 검증구간의 잔차 분포를 기준으로 보정한다.

검증구간에 없던 신규 생산 패턴, 라인 변경, 계절 변화, 제품 mix 변화가 운영구간에 생기면 coverage가 깨질 수 있다.

### 8.6 모델 성능 게이트 부족

현재 테스트에서는 모델 구조, 단조성, 피처 parity 등은 검증하지만, 운영 목적에 직접 연결되는 다음 조건을 강하게 막지는 않는다.

- 공장-대상별 P95 초과율이 과도한지
- PICP_90이 목표보다 낮은지
- 최근 30~60일 coverage가 악화되는지
- 과사용 후보가 특정 항목에 편중되는지

## 9. 목적 적합성 평가

현재 v5.3은 사용량 예측과 예측구간 산출 모델로는 방향이 맞다. 하지만 과사용 일자를 관리자에게 보고받기 위한 이상 탐지 모델로 쓰기에는 후처리 기준이 부족하다.

정확히 말하면 현재 모델은 다음 역할에 적합하다.

- 예상 사용량 P50 산출
- 조건부 정상범위 후보 P05~P95 산출
- 과사용 의심 후보 1차 필터링

하지만 다음 역할을 단독 수행하기에는 부족하다.

- 실제 비정상 과사용 확정
- 관리자 보고가 필요한 이벤트 선별
- 정상 고부하일과 이상 과사용일 구분

따라서 P95 초과를 최종 이상으로 쓰기보다, 추가 규칙과 운영 피드백을 결합한 2단계 탐지 구조가 필요하다.

## 10. 개선안

### 10.1 P95 초과를 1차 후보로 격하

현재의 `actual > P95` 판정은 유지하되, 화면과 보고서에서는 이를 바로 이상으로 표현하기보다 과사용 의심 후보로 표현하는 것이 좋다.

추천 표현:

```text
AI 예상범위 상단 초과
과사용 의심 후보
관리자 확인 필요
```

피해야 할 표현:

```text
이상 발생
비정상 사용
과사용 확정
```

### 10.2 2단계 경보 기준 도입

P95 초과만으로 알림을 만들지 말고, 다음 조건을 함께 보도록 한다.

- P95 초과 여부
- P95 대비 초과율
- 절대 초과량
- band_position
- 최근 연속 초과 여부
- 같은 날 여러 에너지원 동시 초과 여부
- 생산량 또는 재공품 급증 여부
- 휴일, 연휴 직후, 세척 가능일 여부

예시:

```text
관심: actual > P95
주의: actual > P95 and 초과율 >= 5%
경고: actual > P95 and 초과율 >= 10% and 절대 초과량 기준 충족
보고요청: 경고 조건이 2일 이상 연속 또는 동일일 복수 항목 동시 발생
```

### 10.3 P98 또는 P99 상한 추가

관리자 보고 요청 기준은 P95보다 더 높은 분위수를 쓰는 것이 적절할 수 있다.

권장 구조:

- P05~P95: 일반 정상범위 시각화
- P95 초과: 관찰 후보
- P98 또는 P99 초과: 관리자 확인 요청 후보

이를 위해 모델을 P05/P50/P95에 더해 P98 또는 P99까지 학습하거나, CQR 기반으로 P95 상단을 별도 확장한 운영 상한을 둘 수 있다.

### 10.4 상단 전용 보정 도입

과사용 탐지 목적이라면 상단 coverage를 별도로 맞춰야 한다.

현재:

```text
P05 = P05 - q_hat
P95 = P95 + q_hat
```

개선:

```text
P05 = P05 - q_hat_lower
P95 = P95 + q_hat_upper
```

특히 P95 초과율이 높은 항목은 `q_hat_upper`를 더 크게 잡아야 한다.

### 10.5 공장-대상별 calibration layer 추가

모델을 재학습하지 않더라도 최근 운영 이력으로 상단 보정을 추가할 수 있다.

예:

```text
upper_calibration_factor[factory, target]
upper_additive_margin[factory, target]
```

최근 60~90일 기준 P95 초과율이 목표보다 높으면 상한을 넓히고, 너무 낮으면 상한을 줄이는 방식이다.

우선 적용 대상:

- 김해 연료
- 광주 용수
- 남양주2 용수
- 김해 전력
- 논산 연료

### 10.6 배포 전 coverage gate 추가

새 모델을 활성화하기 전 다음 조건을 검사하는 게 좋다.

```text
공장-대상별 PICP_90 >= 0.85
공장-대상별 P95 초과율 <= 8%
전체 PICP_90 >= 0.88
최근 테스트 구간에서 특정 항목 P95 초과율 10% 초과 시 경고
```

이 기준을 만족하지 못하면 모델 활성화는 가능하더라도 “운영 밴드 보정 필요” 상태로 표시해야 한다.

### 10.7 운영 피드백 라벨 저장

관리자가 확인한 결과를 다음과 같이 라벨링해 저장해야 한다.

- 정상 고부하
- 생산 증가로 설명됨
- 재공품 또는 제품 mix로 설명됨
- 세척/CIP/재가동 영향
- 설비 이상
- 계측 이상
- 원인 미확인

장기적으로는 이 라벨을 활용해 단순 분위수 초과가 아니라 실제 보고 필요 이벤트를 분류하는 2차 모델을 만들 수 있다.

### 10.8 최근 drift 모니터링 강화

현재 모니터링은 주로 P50 예측 오차와 bias 중심이다. 여기에 밴드 품질 지표를 추가하는 것이 좋다.

추가 지표:

- 최근 30일 P95 초과율
- 최근 30일 P05 미만율
- 최근 30일 PICP
- 공장-대상별 band_position 평균
- 상단 초과의 연속성

## 11. 최종 판단

v5.3 모델은 에너지 사용량의 조건부 예측범위를 산출하는 모델로는 설계 방향이 타당하다. P05/P50/P95 분위수 회귀와 CQR 보정도 논리적으로 맞는 접근이다.

다만 현재 운영 목적이 “과사용 일자 식별 및 관리자 이상원인 보고”라면, P95 초과를 최종 이상으로 해석하는 것은 과민하다.

정상 가동 상태에서도 P95 초과가 발생하는 이유는 다음 세 가지가 겹친 결과다.

1. P95 자체가 정상 상위 5%를 허용하는 통계 경계다.
2. 하루 15개 항목을 동시에 보므로 하나 이상 초과가 자주 발생한다.
3. 일부 공장-대상은 실제 P95 초과율이 10% 이상으로, 밴드 보정이 부족하다.

따라서 개선 방향은 모델을 완전히 바꾸는 것보다 다음 순서가 현실적이다.

1. P95 초과를 1차 후보로 재정의한다.
2. 초과율, 절대량, 연속성, band_position을 조합한 2단계 경보 기준을 만든다.
3. 공장-대상별 상단 보정을 추가한다.
4. 운영 이력 기반 coverage gate를 만든다.
5. 관리자 피드백 라벨을 저장해 실제 이상 탐지 모델로 확장한다.

이렇게 바꾸면 현재 모델의 예측구간 장점은 유지하면서, 정상 고부하일을 과도하게 이상으로 잡는 문제를 줄일 수 있다.
