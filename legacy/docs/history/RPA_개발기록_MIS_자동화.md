> **기술 스택**: Python, pywinauto, openpyxl, win32clipboard

## 📌 1. 프로젝트 배경

### 기존 업무 방식 (수동)

매일 아침 에너지 담당자가 다음 작업을 **5개 공장 × 반복** 수행:

```
① MIS 프로그램 실행
② "유틸리티 일자별 사용량 추이" 화면 진입
③ 공장 필터 선택 (남양주1 → 남양주2 → 김해 → 광주 → 논산)
④ 기준년월 입력 (예: 2026-05)
⑤ 조회 버튼 클릭
⑥ 그리드 데이터 복사
⑦ Excel 파일 열기 → 해당 시트에 붙여넣기
⑧ ③~⑦ 반복 (5회)
```

- ⏱ **소요 시간**: 약 15~20분/회
- 😩 **문제점**: 단순 반복 작업, 실수 가능성, 매일 출근 직후 수행 필요

### 🔹 STEP 2: "버튼을 찾을 수 없다" — UI 인식 불가 문제

**문제 발생** 🔴  
일반적인 자동화 프로그램은 화면의 버튼을 "이름"으로 찾습니다.

```python
# 예: "조회" 버튼을 이름으로 찾아 클릭
button = window.child_window(title="조회", control_type="Button")
button.click()
```

그런데 MIS 프로그램은 **커스텀 렌더링 UI**를 사용하고 있었습니다.  
쉽게 말하면, 화면에 보이는 버튼들이 실제로는 **"그림"에 가깝고**,  
프로그램 내부적으로 "이것이 버튼이다"라는 정보를 제공하지 않았습니다.

**진단 스크립트 결과**:

```
ComboBox/Edit/List 타입 컨트롤 검색 중...
(아무것도 없음!)          ← 버튼, 입력창, 리스트 전부 인식 불가
```

**해결** ✅  
→ 이름 기반 탐색을 포기하고, **화면 좌표 기반 클릭**으로 전환  
→ 좌표를 `utility_coords.json` 파일에 분리하여, 코드 수정 없이 좌표만 조정 가능하게 설계

```json
{
  "coords": {
    "factory_dropdown": [417, 106],
    "query_button": [328, 66],
    "copy_button": [332, 725]
  }
}
```

### 🔹 STEP 3: "공장이 선택 안 된다" — 드롭다운 조작 문제

**문제 발생** 🔴  
공장 필터 드롭다운을 클릭한 후, F1A(남양주1)를 선택하려 했으나 실패.

**원인 분석**:  
코드가 드롭다운을 열고 → 리스트 항목을 검색하는 중간에  
`set_focus()` (창 활성화)를 호출하면서 **드롭다운이 닫혀버리는** 현상 발생.

```
드롭다운 클릭 → ✅ 열림
└→ 리스트 검색 실패 → set_focus() 호출 → ❌ 드롭다운 닫힘!
   └→ 좌표 클릭 → ❌ 이미 닫혀서 빈 화면 클릭
```

**해결** ✅  
→ 중간 단계(검색, 포커스)를 모두 제거  
→ **"클릭 → 대기 → 클릭"** 단순 흐름으로 변경

```python
# 1. 드롭다운 클릭
window.click_input(coords=(417, 106))
time.sleep(2.0)  # 열릴 때까지 대기

# 2. 바로 F1A 항목 클릭 (아무것도 하지 않고!)
window.click_input(coords=(417, 167))
```

### 🔹 STEP 4: "클립보드가 비어있다" — 데이터 복사 문제

**문제 발생** 🔴  
MIS에서 복사 버튼 클릭 후 Python에서 클립보드를 읽으면 항상 비어있음.

**시도한 방법들**:

| 시도 | 방법                          |   결과   |
| :--: | :---------------------------- | :------: |
| 1차  | `ctypes` (Windows 저수준 API) | ❌ 빈 값 |
| 2차  | `pyperclip` 라이브러리        | ❌ 빈 값 |
| 3차  | PowerShell `Get-Clipboard`    | ❌ 빈 값 |
| 4차  | `pandas.read_clipboard()`     | ❌ 빈 값 |

**원인 발견** 🔍  
클립보드 포맷 진단 스크립트를 작성하여 분석한 결과:

```
클립보드에 3개 포맷 발견:
  포맷 49161 : DataObject
  포맷 49844 : Csv          ← MIS 전용 포맷!
  포맷 49171 : Ole Private Data

★ = 텍스트 포맷 (Python으로 읽기 가능)
   → 텍스트 포맷이 없음!
```

MIS가 표준 텍스트가 아닌 **`Csv`라는 자체 포맷**으로만 데이터를 저장하고 있었습니다.  
일반 클립보드 읽기 함수는 텍스트(CF_TEXT) 포맷만 읽으므로, 데이터를 찾지 못한 것입니다.

**해결** ✅  
→ `win32clipboard`의 `RegisterClipboardFormat("Csv")`로 MIS 전용 포맷을 등록하고 직접 읽기

```python
import win32clipboard

csv_fmt = win32clipboard.RegisterClipboardFormat("Csv")
win32clipboard.OpenClipboard()
data = win32clipboard.GetClipboardData(csv_fmt)  # MIS 전용 포맷으로 읽기!
text = data.decode("utf-8")
win32clipboard.CloseClipboard()
```

### 🔹 STEP 5: "숫자가 깨진다" — CSV 파싱 문제

**문제 발생** 🔴  
데이터는 읽혔으나, 숫자가 이상하게 파싱됨:

| 원래 값 | 파싱 결과 |
| :-----: | :-------: |
|  4,860  |     4     |
|  3,268  |    860    |
|  3,993  |     3     |

**원인**: 데이터 구분자가 쉼표(`,`)인데, 숫자의 천단위 구분기호도 쉼표(`,`)  
→ `4,860`이 `4`와 `860` 두 개로 분리됨

**해결** ✅  
→ Python 표준 `csv.reader` 모듈 사용 (인용부호 안의 쉼표를 구분자로 취급하지 않음)

```python
import csv
reader = csv.reader(io.StringIO(raw_text), delimiter=",")
# "4,860" → 하나의 값으로 정상 인식
```

### 🔹 STEP 6: "기존 데이터에 덮어쓰기가 안 된다" — 날짜 비교 문제

**문제 발생** 🔴  
5월 1일~12일 데이터가 이미 있는데, 새로 조회해도 덮어쓰지 않고 계속 옆에 추가됨.

**원인**: Excel 셀에 저장된 날짜의 **실제 값과 표시 형식이 다름**

```
표시: '26-05-01  (사람 눈에 보이는 형태)
실제 값: datetime(2026, 5, 1)  (프로그램이 읽는 형태)
```

파서가 생성한 문자열 `"26-05-01"`과 Excel의 `datetime(2026, 5, 1)` 객체는  
문자열 비교 시 **항상 불일치** → 새 열로 추가됨.

**해결** ✅  
→ Excel 셀 값이 날짜 객체이면 동일한 문자열 형식으로 변환 후 비교

```python
if isinstance(cell_val, datetime):
    normalized = cell_val.strftime("%y-%m-%d")  # → "26-05-01"
# 이제 파서의 "26-05-01"과 일치!
```

## 📌 4. 핵심 교훈 (기술적 인사이트)

### 💡 "표준이 아닌 프로그램"에 대응하기

| 일반 프로그램            | MIS (커스텀 UI)           |
| :----------------------- | :------------------------ |
| 버튼 이름으로 찾기 가능  | 화면 좌표로만 조작 가능   |
| 클립보드에 텍스트로 복사 | 전용 포맷(Csv)으로만 복사 |
| 표준 단축키 동작         | 일부 단축키 미지원        |

→ **핵심**: 자동화 대상의 특성을 먼저 진단(디버깅)하고, 거기에 맞는 방법을 선택해야 합니다.

### 💡 "단순할수록 안정적이다"

처음에는 여러 단계의 fallback 로직을 구현했지만,  
오히려 중간 과정이 부작용을 일으키는 경우가 있었습니다.  
(예: `set_focus()`가 드롭다운을 닫아버리는 문제)

최종적으로 **"클릭 → 대기 → 클릭"** 의 단순한 흐름이 가장 안정적이었습니다.
