남양주과 전사를 가상 집계 공장으로 추가했고, ai_prediction.py, usage_prediction_v5_service.py, v5_common.py 에서 함께 처리되도록 연결했습니다. 집계 모드에서는 저장된 개별 공장 예측값을 먼저 재사용하고, 없는 날짜/타깃만 각 공장별로 추가 예측한 뒤 합산합니다. 남양주은 남양주1+남양주2, 전사는 남양주1+남양주2+김해+광주+논산 합계로 보입니다.

동작은 이렇게 바뀌었습니다.

예측 실행 탭: 공장 선택에 남양주, 전사가 추가되었습니다.
집계 공장 선택 시: 수동 입력 에디터는 숨기고, 구성 공장 기본 입력값으로 자동 계산합니다.
이력 조회 탭: 남양주, 전사 필터로 집계 조회가 가능합니다.
집계 조회/실행 시: 이미 있는 개별 공장 예측은 재사용하고, 없는 것만 추가 예측합니다.
중요한 운영 포인트는 하나 있습니다. 이번 구현은 집계 결과를 별도 공장 row로 DB에 저장하지 않고, 개별 공장 예측을 기반으로 화면에서 합산하는 방식입니다. 장점은 안전하고 스키마 변경이 없다는 점이고, 단점은 남양주/전사 조회를 자주 하면 그때그때 합산 로직이 돈다는 점입니다.

검증은 .venv 파이썬으로 py_compile까지 통과했고, 실DB를 붙여 실제 UI 클릭까지는 아직 실행하지 않았습니다.

더 좋은 운영안도 같이 드리면:

현재 적용안: 화면에서 실시간 집계. 구현 리스크가 가장 낮고 바로 운영 가능
개선안: prediction_log_aggregate 같은 집계 캐시 테이블을 따로 둬서 남양주/전사를 미리 저장. 조회 속도와 일관성은 더 좋지만, DB 스키마 추가와 캐시 갱신 규칙이 필요
제 추천은 일단 지금 적용한 방식으로 써보고, 사용 빈도가 높아 속도나 일관성 요구가 커지면 그때 집계 캐시 테이블로 가는 겁니다.
