## 1. 단기 (1~2주 · 저위험 · 고효과)

### 1-1. 침묵 실패(silent failure) 제거 — *모니터링 시스템 신뢰성의 핵심*
- **문제**: `_fetch_anomaly_data`/`_fetch_spc_data`(dashboard_main), `get_targets`(target_service), `list_events`(event_annotation_service) 등이 `except Exception: return 빈DF` 패턴 → **DB 장애 시 대시보드가 "✅ AI 이상감지: 정상" 녹색 배너를 표시**. 기동 시 자동 동기화 실패도 stdout `print`로만 남고, footer의 "데이터 수집 엔진 정상 작동 중"·"최근 동기화" 시각은 실제 상태와 무관한 하드코딩/렌더 시각.
- **제언**: ① `logging` 파일 핸들러 도입(print 전면 대체), ② '데이터 없음'과 '조회 실패'를 구분하는 반환 규약(None vs 빈 DF), ③ footer 상태를 실제 마지막 sync 결과(상태 JSON/DB MAX(updated_at))로 표시, ④ 실패 시 사이드바 경고 배지.

### 1-2. DB 액세스 레이어 일원화 (커넥션 풀 + 컨텍스트 매니저)
- **문제**: 17개 파일 ~50개 지점이 호출마다 새 TCP 커넥션 생성/해제, `conn=get_connection()/try/finally` 보일러플레이트 30회+ 반복. 커넥션 생성마다 `is_admin()`→소켓/DNS 조회 동반. 대시보드 1회 렌더에 수십 커넥션.
- **제언**: `mysql.connector.pooling.MySQLConnectionPool`을 admin/viewer 2개 풀로 구성, `with db.cursor() as cur:` 스타일 컨텍스트 매니저 제공, `is_admin` 결과는 rerun 단위 메모이즈. 업로드 경로의 행 단위 N+1 UPSERT(upload_service, daily_energy_sync)는 production_dw_sync가 이미 쓰는 `executemany` 배치 패턴으로 전환.

### 1-3. 도메인 상수 단일 소스화 (`app/domain/factories.py`)
- **문제**: 공장 목록·표시순서·상징색·`남양주=남양주1+남양주2` 집계 규칙이 **최소 9곳**(dashboard_main 5곳, excel_parser, query_service, production_dw_service, v5_common 등)에 분산 하드코딩. 에너지 계열(남양주1/2 분리)과 생산 계열(남양주 통합)의 코드 체계 차이가 파일마다 암묵적.
- **제언**: 공장 마스터(실공장, 집계 그룹, F-코드 매핑, 표시순서, 색상)를 단일 모듈로 정의하고 화면·서비스는 파생 리스트만 사용. 원단위 재계산 람다(8곳 복제)도 `recalc_unit_rates(df)` 공용 헬퍼로 통합 — **"원단위 = Σ사용량/Σ생산톤" 핵심 경영지표 정의가 한 곳에 있어야 함**.

### 1-4. 저장소 위생 자동화 (pre-commit)
- **제언**: ① `gitleaks`(시크릿 차단), ② 1MB 이상 바이너리 커밋 차단, ③ `ruff --select F401,F841`(미사용 import/변수 — 이번에 발견된 죽은 코드류 재발 방지). 에디터의 atomic-write 임시파일(`*.py.tmp.*` 117개 발생원) 설정도 점검.

### 1-5. 외부 절대경로(`E:\Sampled DB\...`)의 환경변수화
- **문제**: 원천 데이터 경로가 **최소 6개 파일**(v5_common, production_dw_service, RPA 3종, WIP_refactoring)에 하드코딩. 드라이브 변경·PC 교체 시 silent 실패.
- **제언**: `.env`에 `SAMPLED_DB_DIR` 1개를 정의하고 공용 config 모듈에서 1회 해석. RPA 도구와 앱 서비스가 같은 상수를 공유.

## 2. 중기 (1~2개월 · 구조 개선)

### 2-1. v5 예측 파이프라인 — 학습/추론 코드 단일화 ★최우선 구조 과제
- **문제(실제 발산 3건 확인)**: 같은 v5.1 모델을 만드는 코드가 2벌(modeling_v5.1.py / v5_retrain_worker.py)인데 ① worker의 M1 피처 토큰에서 캘린더 피처 7종 누락(1차 수정 적용함 — 근본 해법은 단일화), ② 근로자의 날(5/1) 휴일 보강이 학습(modeling_v5.3)에는 없고 추론(v5_common)에는 있는 train/serve skew, ③ TRAIN_START 2021 vs 2023 발산. pkl을 해석하는 추론 코드도 4벌(운영 서비스 + 연구 스크립트 3종).
- **제언**: `select_features`·휴일 로딩·TRAIN_START·추론 함수를 v5_common(또는 `app/ml/`)으로 이관하고, **"동일 입력 날짜에 대해 학습 스크립트와 추론 서비스가 동일한 피처 행렬을 생성하는지" 자동 패리티 테스트** 추가. PLAN.md에 기록된 과거 사고(오프라인 평가 vs 웹 예측 7~64배 괴리)가 바로 이 이중화에서 발생.

### 2-2. 모델 아티팩트 거버넌스 (2.6GB pkl 문제)
- **문제**: ① 재학습이 같은 경로 pkl을 `os.replace`로 덮어쓰는데 `st.cache_resource` 키가 경로 문자열뿐 → **재학습 후에도 실행 중인 앱은 구버전 모델을 계속 서빙**(stale cache). ② 모델 전환 시 v5.2+v5.3 동시 적재로 RAM ~5GB(OOM 위험). ③ 레지스트리의 `v5` 항목은 pkl이 없는 dangling(롤백 선택 시 실패), v5.2는 학습 스크립트가 소실되어 재현 불가. ④ 활성 v5.3의 학습 시점 성능표 부재.
- **제언**: ① 모델 파일명 버전드(`v5.3_YYYYMMDD.pkl`) + 레지스트리 포인터 갱신 방식으로 핫스왑, 캐시 키에 mtime 포함, `max_entries=1`. ② `joblib.dump(compress=3)` + early stopping으로 크기 축소, 공장 단위 분할 lazy 로딩 검토. ③ 레지스트리에 {metrics, split, git commit, SHA-256, 학습데이터 말일} 기록 — 무결성 검증(pickle 임의코드실행 방어)과 모델 교체 의사결정 근거 확보. ④ 보존 정책(활성+직전 1세대) 수립 — 현재 pkl 누적 ~6.4GB, `_archive`의 966MB bak은 외부 보관 후 삭제 권장.

### 2-3. 추론 입력의 엑셀 의존 해소 (DB 이관)
- **문제**: 예측 핵심 입력 3종이 모두 엑셀 — 날씨(DB_weather.xlsx, **비원자적 쓰기 + 동시 접근 보호 없음** → 동시 예측 시 파일 손상 가능), WIP(외부 드라이브 절대경로), 휴일(호출마다 재파싱). 날씨 동기화(API 4회+엑셀 전체 재작성)가 첫 예측 요청을 인라인 블로킹.
- **제언**: MySQL이 이미 운영 중이므로 `weather_daily`/`wip_daily`/`holiday` 테이블로 이관. 단기 완화책: 엑셀 쓰기를 임시파일+`os.replace` 원자적 교체로, 동기화는 백그라운드/스케줄러로 분리.

### 2-4. 재학습 오케스트레이션 견고화
- **문제**: 락 파일에 트리거(Streamlit) PID를 기록하고 해제는 워커 finally에만 의존 → 워커가 import 단계에서 죽으면 **24시간 데드락**, 반대로 24시간 초과 학습은 stale 판정되어 **동시 학습 경합**. `get_prediction_history`(조회 함수)가 누락 일자를 만나면 추론+INSERT를 수행하는 숨은 부수효과(조회가 수 분 블로킹).
- **제언**: 락에 워커 PID 기록 + heartbeat(주기적 touch) + PID 생존 확인. 일일 마감 배치(전일 예측 + backfill + 날씨 동기화)를 Windows 작업 스케줄러로 분리해 화면 조회는 항상 읽기 전용으로.

### 2-6. 보안 베이스라인
- **XSS**: `unsafe_allow_html=True` 146회 산재, 사용자 입력(이벤트 메모)·DB 값(audit)·LLM 출력이 escape 없이 DOM 삽입 — stored XSS 경로. → `safe_html()` 공용 헬퍼로 사용자/DB 유래 값 기본 escape, LLM 보고서는 허용 태그 화이트리스트 sanitize(bleach).

## 3. 장기 (분기 단위 · 아키텍처)

### 3-1. 인증 모델 전환
IP 기반(호스트=root) 권한은 ① 리버스 프록시/터널 하나로 전원 관리자 승격, ② Streamlit 비공개 API(`_session_mgr`) 의존으로 버전업 시 조용히 파손되는 구조적 한계. → 단계적으로: 현 전제("프록시 뒤 배포 금지") 문서화 → `streamlit-authenticator` 등 관리자 로그인 도입(viewer는 무인증 유지) → audit 기록을 세션 사용자 기반으로 전환(현재는 모든 변경이 서버 OS 계정으로 기록됨).

### 3-2. `app/predictive model` → `app/predictive_model` 패키지 재편
폴더명 공백 때문에 ① 파이썬 패키지 import 불가 → 전 스크립트가 `sys.path` 핵 사용, ② **import 실패 시 빈 shortlist로 조용히 폴백**(WIP 피처 전부 빠진 모델이 무경고 배포될 수 있는 위험). 운영 모듈(`wip_feature_shortlist.py`)은 `app/services/`로, 연구 스크립트는 `research/`로, 모델 바이너리는 git 트리 밖 `models/`로. 레지스트리 경로·gitignore·서비스 참조 4곳을 한 PR에서 일괄 수정해야 하는 작업이므로 `_pick_existing_dir` 후보 추가 방식의 점진 이행 권장.

### 3-3. 스키마 마이그레이션 체계화
schema.sql + `_PENDING_COLUMN_MIGRATIONS` 하드코딩 + 수동 스크립트의 3원 체제 → 버전 테이블 기반 마이그레이션 러너(또는 Alembic)로 통일. `init_db`의 매 rerun DDL 문제도 구조적으로 해소(1차로 `cache_resource` 가드 적용됨).

### 3-4. RPA 견고화
좌표 JSON에 schema_version/캡처일/MIS 버전 메타 추가(화면 변경 감지), 클립보드 폴링 적응형 타임아웃, 좌표 측정 도구(mouse_*.py)는 dev 폴더로 분리, 로그 디렉터리를 `logs/automation/`으로 통일(현재 3곳 분산), 일회성 스크립트(`rerun_may_predictions.py`)는 CLI 인자형 범용 스크립트로 전환.

### 3-5. 운영 데이터·계수의 외부화
- 드리프트/휴일 보정계수(`DRIFT_CORRECTIONS` 등)와 WIP 환산계수(260014×10.91954 등)는 사실상 **경영지표 정의**인데 코드 상수로 존재 → DB 테이블(적용기간·근거·변경자) 또는 설정 파일로 이관해 변경을 audit과 연결. 특히 v5.3은 캘린더 피처를 이미 학습에 포함하므로 휴일 보정과 **이중 보정 위험** — 만료 조건 명시 필수.
- `validate_data`의 결측→0 치환은 '미보고'와 '실제 0'을 구분 불가하게 만들어 원단위 분모 왜곡 + 학습 데이터 오염 → NULL 유지 또는 치환 건수 경고 리포트.

### 3-6. 품질 인프라
테스트가 1개뿐. 우선순위: ① 피처 패리티 테스트(2-1), ② excel_parser/validation 회귀 테스트(업로드는 운영 핵심 경로), ③ 원단위 가중평균 규칙 테스트, ④ query_service 집계(전사/남양주 합산) 테스트. 문서-코드 정합성도 자동 점검(존재하지 않는 modeling_v5.2.py를 안내하는 문서 등 — `project_FEMS_info.md`·`RUN_GUIDE_KR.md`·`AGENT_GUIDE_KR.md`의 v5.3 기준 현행화 필요).

| 위치 | 내용 | 위험도 |
|------|------|--------|
| `production_performance.py` `_energy_cross` | 공장 필터 미적용 — 김해만 선택해도 에너지 라인은 전사 값(차트 왜곡) | 중 |
| `ai_prediction.py` 일괄 예측 | 결과가 session_state 비영속 — 위젯 터치 시 수십 초 작업 유실 | 중 |
| `audit_log.py` | LIMIT 2000 후 pandas 필터 — 초과 기간 통계 왜곡 / 행당 위젯 폭증 | 저 |
| `wip_daily_rpa.py` | MIN_SPAN_DAYS 월초 로직이 전월 재공품 혼입 가능(의도 불명) | 저 |
| `aggregate_to_monthly`(pandas 경로) | 계획 dedup 키에 factory 누락 — SQL 경로와 달성률 불일치 가능(레거시 경로 한정) | 저 |
| `_cached_current_user` | 전 세션 공유 캐시 + 서버 OS 계정 — 다중 관리자 도입 시 감사 추적 무의미 | 저(설계 제약) |

> 이 항목들은 동작 변경(수치/표시 의미가 바뀜)을 수반하므로 운영 사용자 공지와 함께 별도 적용을 권장.
