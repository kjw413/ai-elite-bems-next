# v5 예측 모델 누수 제거 및 재학습 복구

## Summary
- 저장소 기준으로 원인을 확인했습니다. 공용 피처 생성 로직이 현재일 `intensity`를 만들고, 훈련 경로 두 곳이 `select_dtypes()`로 모든 숫자 컬럼을 X에 넣고 있으며, 추론 경로는 없는 피처를 `0`으로 채워 학습/추론 분포가 크게 어긋납니다.
- 현재 활성 `v5.pkl`에도 `전력원단위_kWh_mix_ton_`, `연료원단위_N__mix_ton_` 같은 누수 피처가 실제로 저장돼 있습니다.
- 별도 문제로, 백그라운드 재학습은 레지스트리에 `active_model_path`가 없어 이미 실패 중입니다. 누수 수정과 함께 재학습 부팅 경로도 같이 고쳐야 합니다.
- 목표는 “예측 시점에 실제로 알 수 있는 정보만 쓰는 피처 명세”로 통일하고, 기존 누수 모델은 강제로 교체되도록 만들어 웹 예측과 오프라인 성능표가 같은 조건에서 맞도록 하는 것입니다.

## Key Changes
- 공용 피처 명세를 [v5_common.py](e:/AI-Elite_Energy-Dashboard-Web/app/services/v5_common.py)로 단일화합니다.
- `make_features()`는 예측 시점 안전 피처만 반환하도록 바꿉니다.
- 유지할 피처는 현재일 `mix_prod_kg`/기상/달력 정보와, 타겟의 과거 기반 피처(`lag1`, `r7mean`, `intensity_lag1`) 및 기상 파생지수(`HDD`, `CDD`, `THI`)로 제한합니다.
- `intensity_lag1`는 `현재일 intensity`를 만든 뒤 shift 하는 방식이 아니라, `shift(1)` 된 타겟과 생산량으로 직접 계산해 현재일 실측값이 중간에도 생기지 않게 합니다.
- 모든 `*원단위*`, 현재일 `intensity`, 다른 타겟의 현재일 실측치, `freezing_power_kwh`/`air_compressor_kwh`/`wastewater_ton` 같은 현재일 계측값은 X에 절대 들어가지 않도록 “블랙리스트”가 아니라 “허용 목록” 방식으로 막습니다.
- 전역 `dropna()`는 없애고, 실제 모델 입력에 필요한 컬럼만 기준으로 결측 제거하도록 바꿔 예측행이 무관한 NaN 때문에 탈락하지 않게 합니다.
- 백그라운드 재학습 로직은 [v5_retrain_worker.py](e:/AI-Elite_Energy-Dashboard-Web/app/services/v5_retrain_worker.py)에서도 같은 공용 피처 명세를 쓰게 맞춥니다.
- 저장 모델 스펙에 `feature_spec_version`을 추가하고, 기존 모델에 이 값이 없거나 금지 피처가 섞여 있으면 “구형 누수 모델”로 판정해 weight update를 건너뛰고 즉시 full retrain 하도록 바꿉니다.
- 추론 서비스 [usage_prediction_v5_service.py](e:/AI-Elite_Energy-Dashboard-Web/app/services/usage_prediction_v5_service.py)에서는 `reindex(fill_value=0.0)`로 숨기지 말고, 필요한 피처가 없거나 구형 모델이면 명시적으로 “모델 재생성 필요” 오류를 내도록 바꿉니다.
- `energy_daily`에 누수 컬럼이 있어도 예측용 조회에는 다시 포함하지 않습니다. 이번 수정의 핵심은 DB 컬럼 확장이 아니라 안전 피처만 쓰는 구조로 고정하는 것입니다.
- 레지스트리가 없어도 기존 `v5.pkl`이 있으면 이를 기본 active model로 인식해 재학습이 시작되게 하고, 성공 시 `v5_model_registry.json`을 반드시 기록하도록 정리합니다.
- 단독 실행용 훈련 스크립트도 같은 공용 피처 규칙과 메타데이터를 쓰도록 맞춰, 수동 생성 모델과 웹 재학습 모델이 다시 어긋나지 않게 합니다.

## Interface Changes
- 저장 모델의 각 타겟 스펙에 `feature_spec_version`이 추가됩니다.
- 새 모델의 `features` 목록은 예측 시점 안전 피처만 포함합니다.
- 구형 `v5.pkl`로 예측 요청이 들어오면, 조용히 0 채움 예측을 하지 않고 “재학습/재생성 필요” 상태로 실패합니다.
- 백그라운드 재학습은 레지스트리 파일이 없어도 기존 모델 파일이 있으면 복구 가능한 상태로 동작합니다.

## Test Plan
- 내장 `unittest` 기준의 회귀 테스트를 추가합니다.
- 테스트 1: 피처 생성 결과에 `*원단위*`, 현재일 `intensity`, 다른 타겟 현재일 실측 컬럼이 포함되지 않는지 검증합니다.
- 테스트 2: 최소한의 과거 영업일 이력과 현재일 입력만으로 예측행이 정상 생성되고 필수 피처가 모두 채워지는지 검증합니다.
- 테스트 3: `전력원단위_kWh_mix_ton_` 같은 기존 누수 피처가 저장된 모델 스펙을 legacy/incompatible로 정확히 잡아내는지 검증합니다.
- 수동 검증 1: 새 `v5.pkl` 생성 후 저장된 `features` 목록에 누수 피처가 없는지 확인합니다.
- 수동 검증 2: 김해/논산 기준 2026-01-01~2026-03-31 배치 예측을 다시 돌려, 절대 MAPE가 예전보다 올라가더라도 오프라인 평가와 웹 경로 결과가 더 이상 7~64배씩 벌어지지 않는지 확인합니다.
- 수동 검증 3: 레지스트리 없는 상태에서 재학습을 트리거해 `active_model_path is missing` 오류 없이 성공 상태로 끝나는지 확인합니다.

## Assumptions
- 현재일 `mix_prod_kg`와 기상값은 예측 시점에 합법적으로 알 수 있는 입력으로 간주하고 유지합니다.
- 이번 작업의 기본 원칙은 정확성 우선입니다. 누수 모델은 경고 후 계속 쓰지 않고, 새 모델 생성 전까지 막는 쪽으로 처리합니다.
- 이번 범위는 피처 명세 교정, 훈련/추론 동기화, 재학습 복구, 검증 체계 마련까지입니다. 모델 종류나 하이퍼파라미터 튜닝은 이번 패스에서 건드리지 않습니다.
- 리포트에 적힌 `diagnose_web_vs_training.py`, `eval_offline_april.py` 등은 현재 저장소에 없으므로, 구현 시에는 저장소 내부 테스트/검증 코드로 대체합니다.
