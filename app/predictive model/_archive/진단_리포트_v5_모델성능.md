# FEMS v5 에너지 사용량 예측 모델 성능 진단 리포트

**작성일**: 2026-04-22  
**대상**: 자체제작 FEMS의 v5 앙상블 예측 모델 (`app/predictive model/modeling_v5.py`, `app/services/usage_prediction_v5_service.py`)  
**진단 목적**: 훈련 시 측정된 MAPE(성능표)와 웹 UI에서의 실제 예측 MAPE가 크게 차이나는 원인을 규명하고 해결 방안을 도출

---

## 1. 한 줄 요약

> **훈련 성능표의 MAPE 2~5%는 데이터 누수(Data Leakage)에 의한 비정상적으로 낮은 값이며, 웹 UI에서 관찰된 17~42% MAPE가 모델의 실제 성능에 가깝다.**

훈련 시 피처(X)에 **"예측해야 할 당일의 실측치로부터 계산된 값"** 과 **"예측 시점에 알 수 없는 다른 실측치"** 가 포함되어 있어, 모델이 사실상 "답을 보고 답을 맞추는" 상태로 학습되었습니다. 웹 예측 경로에서는 이 피처들이 DB에서 조회되지 않아 0으로 채워지면서, 훈련과 예측 사이에 **피처 분포 불일치(feature skew)** 가 발생해 성능이 급격히 악화됩니다.

## 3. 진단 절차 및 결과

### Step 1. 프로젝트 및 알고리즘 파악

- **프로젝트**: Factory Energy Management System. 5개 공장(남양주1/남양주2/김해/광주/논산)의 전력/연료/용수 사용량 예측
- **알고리즘**:
  - 데이터 분할: Train(2021-01~2024-12) / Valid(2025) / Test(2026-01~03)
  - 모델: `LightGBM + XGBoost + CatBoost` × 피처셋 4종(M1~M4) = **12개 앙상블**
  - 피처: 생산량 로그변환, HDD/CDD/THI 기상지수, lag/rolling 변수, 공휴일 거리, intensity(원단위)
  - Target 변환: `log1p / expm1` 사용
  - 이상치 처리: `special_event` (생산량이 30일 rolling median의 10% 미만인 날) 제거
  - 앙상블 가중치: SLSQP로 Valid MAPE 최소화

### Step 2. 성능표(MAPE) 재현성 검증 (실험 A)

훈련 스크립트를 재실행해서 성능표와 동일한 MAPE가 재현되는지 확인.

**결과**: 평균 절대차 **0.20%p**, 최대 차이 **0.79%p** — 성능표와 거의 완벽하게 일치.

| 공장    | 타겟 | 성능표 MAPE | 재현 MAPE | 차이         |
| ------- | ---- | ----------- | --------- | ------------ |
| 남양주1 | 용수 | 2.91%       | 2.91%     | 0.00         |
| 논산    | 용수 | 0.66%       | 0.65%     | -0.01        |
| 남양주1 | 연료 | 3.92%       | 3.97%     | +0.05        |
| 광주    | 전력 | 3.72%       | 4.51%     | +0.79 (최대) |

→ **훈련 로직은 성능표를 정확하게 만들어낸다**는 점이 확인됨. 성능표 자체는 훈련 파이프라인이 생성한 값 그대로.

### Step 3. 4월 홀드아웃 오프라인 검증 (실험 B)

같은 훈련 로직으로 Test 기간만 2026-04-01~16으로 바꾸어 예측.

**결과** (정상 사례 12건 평균): **MAPE 2.77%** — 성능표 수준과 유사.  
단, 일부 사례에서 MAPE 240,636% 등 극단적 값 발생.

- 원인: **2026-04-16 실측치가 0인 컬럼** 존재 (김해 연료/용수, 논산 전력) — 데이터 미적재. 이것은 모델이 아닌 **데이터 품질 이슈**.

→ **이 시점까지는 "모델은 정상, 웹 예측 쪽이 뭔가 문제"라고 잘못 결론**을 냈음.

### Step 4. 웹 예측 실패 재현 (사용자 지적 이후 재조사)

사용자가 웹 UI에서 실제로 MAPE 17~42%가 나왔다고 알려줘, **웹 예측 경로와 훈련 경로의 피처 구성 차이**를 직접 파헤침.

**훈련 경로 vs 웹 예측 경로의 데이터 소스 비교**:

| 항목             | 훈련 파이프라인                             | 웹 예측 파이프라인                                                                                 |
| ---------------- | ------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| 과거 데이터 소스 | `E:\Sampled DB\RawDB_에너지.xlsx` 전체 컬럼 | MySQL `energy_daily` 테이블 5개 컬럼만 (`date, mix_prod_kg, total_power_kwh, fuel_nm3, water_ton`) |
| 예측일 입력      | —                                           | `mix_prod_kg` + 5개 기상값 + 3타겟=0                                                               |
| 피처 컬럼 매칭   | `X_all` 자동 추출                           | pkl에 저장된 `features[mt]`로 `reindex(fill_value=0.0)`                                            |

이 차이가 만들어내는 결정적 결과:

#### 결정적 발견 — 훈련 피처 목록 (전력 타겟 예시)

```
전력원단위_kWh_mix_ton_        ← 전력 / 믹스톤 = 당일 실측 전력을 직접 입력!
연료원단위_N__mix_ton_         ← 연료 실측치 기반
용수원단위_ton_mix_ton_        ← 용수 실측치 기반
폐수원단위_ton_mix_ton_        ← 폐수 실측치 기반
intensity                      ← 전력 / 믹스톤 (동일하게 당일 실측 답)
연료량_N__, 용수량_ton_        ← 다른 타겟 실측치 (예측 시 알 수 없음)
냉동전력량_kWh_, 공압기_kWh_   ← 전력의 세부 성분 실측치
폐수량_ton_                    ← 폐수 실측치
```

이 피처들은 M1, M2, M3, M4 **모든 모델에 포함**되어 있음.

- **훈련·Valid·Test에서는 Excel에 다 채워져 있으므로 피처에 실측값이 들어감** → 모델이 답을 보고 학습 → 낮은 MAPE
- **웹 예측에서는 DB에 없으므로 `reindex(..., fill_value=0.0)` 로 전부 0** → 모델 입장에서는 경험한 적 없는 입력 → 예측 대혼란

## 4. 근본 원인 (Root Cause)

### 🔴 Root Cause #1: 데이터 누수 (Data Leakage) — 당일 실측 기반 피처가 X에 포함됨

훈련 시 `make_features()`에서 생성되는 `intensity` 와 RawDB Excel에 이미 계산되어 있는 **`*_원단위` 컬럼들**이 `X_all = d.select_dtypes(include=[np.number])` 로 자동 포함됨. 이 값은 다음과 같이 계산된 **당일 실측 기반 피처**:

```
전력원단위 = 당일 전력량(kWh) / 당일 믹스생산량(ton)
```

즉, 전력 모델이 예측하는 타겟 `전력량[kWh]` 으로부터 거의 직접 계산된 값이 피처로 들어감 → **정답을 알려주고 맞추라는 꼴**.

### 🔴 Root Cause #2: 다른 타겟 실측치가 피처로 포함됨

전력을 예측할 때 `연료량_N__`, `용수량_ton_`, `냉동전력량_kWh_`, `공압기_kWh_`, `폐수량_ton_` 같은 다른 실측치가 피처에 포함됨. 이들 역시 **예측 시점에 알 수 없는 값**.

### 🔴 Root Cause #3: 훈련 경로와 웹 예측 경로의 피처 소스 불일치

- **훈련**: Excel 전체 컬럼 → 부가 컬럼·원단위 컬럼 모두 자동 포함
- **웹 예측(`_fetch_energy_history`)**: DB 5개 컬럼만 조회 → 부가 컬럼 부재 → `reindex(fill_value=0.0)` 로 대체

훈련에서 중요하게 학습된 피처가 예측 시에 전부 0으로 바뀌니 성능이 붕괴.

### 🟡 부가 발견 #1: `make_features`의 `dropna()` 로직

예측행에 없는 부가 컬럼 때문에 NaN이 생길 수 있음. 웹 경로에서는 DB에 부가 컬럼 자체가 없어서 NaN 문제가 안 생기지만, 이는 "문제가 숨겨진 것"일 뿐 정상 동작은 아님.

### 🟡 부가 발견 #2: MAPE 계산의 분모 클리핑

`mape(y_true, y_pred, eps=1.0)` — 실측치가 0 또는 매우 작을 때 MAPE 값이 폭주함. 2026-04-16처럼 **실측치가 0인 누락 데이터**가 있으면 MAPE가 수십만%로 튀는 원인.

### 🟡 부가 발견 #3: v5 훈련/예측 사이 피처 명세 불일치

- `training_path` 가 `features_map[mt]` 를 pkl에 저장
- 예측 시 `X_row_all.reindex(columns=feat_list, fill_value=0.0)` 로 맞춤
- 저장된 `feat_list`에 leak 피처가 들어 있어도 예측 시에는 0만 들어가므로 **모델 가중치 일부가 사실상 사장됨**

## 5. 왜 실험 A에서 "정상 재현"으로 착각했는가?

실험 A의 재현 스크립트는 훈련 스크립트를 거의 그대로 복제한 것이라, 훈련과 **똑같은 leak 상태**로 돌아갔습니다. 즉 "leak이 있는 훈련"과 "leak이 있는 재현"의 MAPE가 일치한 것이지, 실제 예측 성능을 재현한 것이 아닙니다.

사용자가 웹 UI에서 실제로 예측을 돌려본 뒤에야, leak이 없는 진짜 추론 경로의 MAPE(17~42%)가 드러남.

## 6. 증거 요약

### 6.1 훈련 피처에 leak 존재 확인 (`diagnose_web_vs_training.py` 검증)

```
M1: 15개 피처 중 5개가 당일 실측 기반 ⚠️
  - 전력원단위_kWh_mix_ton_   ⚠️ RISKY (당일 전력 실측)
  - 연료원단위_N__mix_ton_    ⚠️ RISKY
  - 용수원단위_ton_mix_ton_   ⚠️ RISKY
  - 폐수원단위_ton_mix_ton_   ⚠️ RISKY
  - intensity_lag1            (전날값이므로 OK)

M2~M4: 모두 동일 4개 원단위 컬럼이 피처로 포함 ⚠️
```

### 6.2 DB 스키마상 해당 컬럼이 누락되어 있음

`_fetch_energy_history` 가 가져오는 컬럼:

```sql
SELECT date, mix_prod_kg, total_power_kwh, fuel_nm3, water_ton
FROM energy_daily
```

→ `power_per_ton_kwh`, `fuel_per_ton_nm3`, `water_per_ton_ton`, `wastewater_per_ton_ton`,
`freezing_power_kwh`, `air_compressor_kwh`, `wastewater_ton` 이 **전부 조회되지 않음**.

### 6.3 피처 재정렬 시 0 채움

`_infer_target` 의 `X_row_all.reindex(columns=feat_list, fill_value=0.0)` 에서 누락 컬럼 전부 0 대체.

| 이슈                           | 심각도     | 영향 범위                  | 해결 우선순위 |
| ------------------------------ | ---------- | -------------------------- | ------------- |
| Data leakage (원단위 피처)     | **치명적** | 전 공장·전 타겟            | 1             |

| `dropna()` 로 예측행 탈락 위험 | 높음       | 로직 자체는 운이 좋아 피함 | 2             |
| MAPE 분모 폭주 (0 실측)        | 중간       | 평가 지표 왜곡             | 3             |

## 8. 해결 방안 (실행 가능한 단계)

### A. 단기 (즉시) — Leak 피처 제거 및 재학습

1. `make_features()` 및 `X_all` 구성 시 다음 컬럼을 **반드시 제외**:
   - 모든 `*원단위*` 컬럼 (`전력원단위`, `연료원단위`, `용수원단위`, `폐수원단위`)
   - `intensity` (단, `intensity_lag1` 은 전날 기반이라 **유지 가능**)
   - 타겟이 아닌 다른 실측치: `냉동전력량`, `공압기`, `폐수량`, 그리고 예측하고자 하는 타겟을 제외한 다른 두 타겟 실측치
2. `modeling_v5.py` 와 `v5_common.make_features` 를 **동일하게** 수정 (훈련·예측 파이프라인 동기화)
3. 전체 플랜트·타겟 재학습 → `v5.pkl` 재생성
4. 성능표 재측정 (현실적으로 MAPE 5~15% 수준 예상)
5. 웹 UI에서 2026-Q1 재예측 → 성능표와 유사하게 나오는지 확인

### B. 중기 — DB 스키마/조회 점검

1. `_fetch_energy_history` 의 `SELECT` 에 **실제 사용되는 피처 컬럼 모두** 포함 (leak 제거 후 남는 안전 피처만)
2. `_to_korean_schema` 의 `fillna(0.0)` 는 누락 대체용이지만, **의미 없는 0을 피처에 넣는 위험**을 인지. 대안: 과거 평균·중앙값·직전값으로 대체 또는 해당 일 예측 거부.

### C. 중기 — 평가 지표 개선

1. MAPE 계산 시 **실측치가 정상 범위 미만인 날짜는 제외** (예: mix_prod_kg < 전월 중앙값의 10% 이면 special_event로 분리)
2. 보조 지표로 SMAPE 또는 MAE·RMSE 병기
3. `dropna()` 전에 예측행이 탈락하지 않도록, 예측 대상 피처만 엄격히 골라 dropna 대상 컬럼을 명시

### D. 장기 — Leakage 자동 탐지 테스트

- CI 체크: 피처 명에 타겟 문자열이 포함되면 경고
- 타겟과의 상관계수가 비정상적으로 높은 피처(>0.99) 자동 감지
- Train/Valid/Test 분리 후 피처 값 분포 비교 (shift 제대로 적용됐는지 확인)

## 11. 맺음말

성능표의 수치는 **"데이터 누수가 있는 상태에서 측정된 이상적인 숫자"** 이며, 웹 UI에서 나온 17~42%가 **모델의 실제 실력**에 가깝습니다. 이는 모델 선택(앙상블 구조·부스팅 알고리즘)의 문제가 아닌 **피처 엔지니어링의 설계 결함**이므로, leak 피처 제거 후 재학습하면 성능표와 웹 예측이 일치된 현실적인 MAPE를 얻을 수 있습니다. 이후 추가적인 성능 고도화(하이퍼파라미터, 모델 확장 등)는 **이 leak 문제를 먼저 제거한 베이스라인 위에서** 진행해야 의미가 있습니다.
